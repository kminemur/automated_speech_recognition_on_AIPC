# 音声認識アプリ アーキテクチャ

## 1. 設計方針

アプリは「GUI/CLI と認識ロジックを分離する」ことを中心方針とする。表示層に依存しない共通エンジンを持つことで、PyQt6 GUI と CLI の両方から同一の認識処理を利用できるようにする。

## 2. コンポーネント構成

### 2.1 `app.py`

- アプリ全体のエントリーポイント
- コマンドライン引数の解釈
- GUI モードと CLI モードの切り替え
- 設定値を `ASRConfig` にまとめて各実行モードへ渡す

### 2.2 `asr_engine.py`

- 音声認識の中核
- マイク入力の取得
- 入力バッファ管理
- WebRTC VAD による発話区間検出
- OpenVINO `WhisperPipeline` による推論
- 認識結果、状態、ログ、音量レベルのコールバック通知

### 2.3 `asr_gui.py`

- PyQt6 ベースのデスクトップ UI
- モデル、デバイス、マイク、言語、タスク、チャンク秒数などの設定 UI
- 認識結果表示
- ログ表示
- Start / Stop / Clear 操作
- エンジンからのイベントを Qt Signal 経由で UI に反映

### 2.4 `realtime_asr.py`

- 既存互換や簡易起動用の CLI ラッパー
- 実処理は `app.py --cli` に委譲

### 2.5 補助ファイル

- `requirements.txt`: Python 依存管理
- `setup.bat`: 仮想環境作成と依存導入
- `run.bat`: 既定モデルや既定デバイスを考慮した起動

## 3. 論理構造

```text
+-------------------+
|   app.py          |
|  - arg parsing    |
|  - mode switch    |
+---------+---------+
          |
          +----------------------+
          |                      |
          v                      v
+-------------------+   +-------------------+
|   asr_gui.py      |   |   CLI mode        |
|  PyQt6 UI         |   |  console output   |
+---------+---------+   +---------+---------+
          |                       |
          +-----------+-----------+
                      |
                      v
             +--------------------+
             |   asr_engine.py    |
             |  audio + VAD + ASR |
             +---------+----------+
                       |
                       v
             +--------------------+
             | WhisperPipeline    |
             | OpenVINO 2026.0    |
             +--------------------+
```

## 4. データフロー

1. ユーザーが GUI または CLI で設定を与えて起動する。
2. `app.py` が引数を解釈して `ASRConfig` を生成する。
3. `RealtimeASREngine` が OpenVINO Whisper モデルをロードする。
4. `sounddevice.InputStream` からマイク音声を取得する。
5. 音声は内部キューに蓄積され、ワーカースレッドが順次処理する。
6. VAD が発話フレームと無音フレームを判定する。
7. 発話終了または最大長到達時にセグメントを確定する。
8. 確定した音声セグメントを `WhisperPipeline.generate()` に渡す。
9. 得られたテキストを軽く正規化し、重複を抑制する。
10. 結果を GUI または CLI に通知して表示する。

## 5. 並行処理モデル

- 音声キャプチャは `sounddevice` のコールバックで実行する
- 推論処理は別ワーカースレッドで実行する
- GUI 更新は Qt Signal を経由してメインスレッドで行う

この分離により、推論が走っている間も UI 操作性を維持しやすくする。

## 6. 主要クラス設計

### 6.1 `ASRConfig`

責務:

- モデルパス
- デバイス
- サンプルレート
- チャンク秒数
- マイク index
- 言語
- タスク
- VAD パラメータ
- キャッシュディレクトリ

目的:

- GUI / CLI / エンジン間の設定受け渡しを統一する

### 6.2 `RealtimeASREngine`

主な責務:

- 設定検証
- Whisper モデルロード
- マイク入力開始・停止
- 音声フレームのバッファリング
- VAD 状態管理
- セグメント確定
- 推論実行
- 結果通知

外部インターフェース:

- `start()`
- `stop()`
- `running`

通知チャネル:

- `on_status`
- `on_log`
- `on_transcript`
- `on_level`

### 6.3 `MainWindow`

主な責務:

- ユーザー設定入力
- エンジン生成と破棄
- 開始停止ボタン制御
- 認識結果表示
- ログ表示
- エラー表示

## 7. 状態遷移

```text
Idle
  -> LoadingModel
  -> Listening
  -> Transcribing
  -> Listening
  -> Stopped

Error は任意状態から遷移しうる
```

補足:

- 実装上は `Transcribing` を明示 state にしなくてもよい
- UI 表示上は `Loading model`、`Listening`、`Stopped`、`Error` を見せると十分

## 8. エラーハンドリング方針

- モデルパス不正: 開始前に検知してユーザーへ通知
- モデルロード失敗: エラー状態へ遷移し詳細ログを出力
- マイク利用不可: 開始失敗として UI/CLI に通知
- 推論中例外: ログ出力後に安全停止、または停止を促す

## 9. 拡張ポイント

## 10. 採用理由

### OpenVINO + WhisperPipeline

- OpenVINO 2026.0 の CPU / GPU / NPU 実行を活かせる
- OpenVINO IR モデルを直接扱える
- Whisper 系モデルを比較的少ない実装量で扱える

### PyQt6

- Windows デスクトップアプリとして扱いやすい
- Signal/Slot によりワーカースレッド連携がしやすい

### sounddevice

- Python からマイク入力をシンプルに扱える

### WebRTC VAD

- 無音区間によるセグメント分割を軽量に行える

## 11. 実装時の注意点

- NPU 利用時はモデル互換性を事前確認する
- `CACHE_DIR` を使ってモデルキャッシュを有効にする
- GUI スレッドから直接推論を呼ばない
- 停止時は InputStream、キュー、ワーカースレッドの順に確実に閉じる
- 認識結果は完全一致重複のみでも最低限抑制しておく
